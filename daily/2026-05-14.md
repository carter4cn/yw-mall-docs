## 🔐 登录改造 P0（feat/login-revamp.md）

**触发动因：** S1-S3 期间反复遇 JWT "signature is invalid"，根因是 user-rpc `onConfigChange` 中 `yaml.Unmarshal` 静默丢 `JwtAuth.AccessSecret` 嵌套字段；e2e 全程靠 `openssl dgst -sha256 -hmac` 手工签 token 兜底。**决策**：不修 JWT，整体替换。

### 架构：Opaque Token + Redis Session

- `access_token` = base64url(32 字节随机) — TTL 1800s
- `refresh_token` = base64url(32 字节随机) — TTL 604800s，每次 refresh 旋转
- `csrf_token` 随 session 下发，写操作走 X-CSRF-Token 双重提交
- `user_sessions:{uid}` SET 索引所有活跃 access_token，改密/踢出一键清

## 🛍️ 晚间 — C 端购物链路打通 + 基础设施优化

> P0 / P0.5 落地后浏览器试用，连环发现一串小坑，集中收尾。

### 1. 购物车真接通后端

**问题**：之前 `addToCart()` 只 `cartStore.increment()` 改本地 ref，没调过 `/api/cart/*`。

**修**：
- 新建 `src/api/cart.ts`（list/add/remove/update-quantity/clear）
- `stores/cart.ts` 重写：CartItem[] + computed count + 全部 await 后端
- `pages/cart/index.vue` 替换占位：列表 + 数量编辑 + 删除 + 全选 + 合计 + 去结算
- `pages/product/detail.vue` addToCart 改 `await cartStore.add(productId, 1)`

### 2. cart-rpc ListItems read-after-write 延迟修复（初版）

**症状**：浏览器加购物车后立刻 list，60% 概率拿不到刚加的项。

**初版修法**：把 SELECT 包到 TransactCtx，ProxySQL pin master。

### 3. nginx upstream IP 缓存修复

**症状**：每次 force-recreate mall-api 后浏览器 502，要 `podman restart mall-fe` 才恢复。

**根因**：nginx `proxy_pass http://mall-api:18888;` 启动时一次性解析 IP，podman 重启换 IP 后不再 re-resolve。

**修**：nginx.conf 加 `resolver 10.89.0.1 valid=30s` + 变量形式 `proxy_pass http://$upstream_api`。

### 4. cart 选择 / 合计 / 去结算 UI + 后端 createOrder 自动默认地址

- cart bottom-bar 加全选圆点 / 单选圆点 / 合计 / 去结算
- 跳坑：bottom-bar 被 tabBar 盖住 → bottom:50px + z-index:10 + safe-area-inset
- `mall-api` CreateOrder 之前透传 AddressId=0，order-rpc 报"收货地址不存在"
- 修：mall-api CreateOrder 先 `GetDefaultAddress(uid)` → 取 addr.Id → 传给 OrderRpc

### 5. 「我的订单」500 修复

**症状**：`/api/order/list` → 500 `not matching destination to scan`

**根因**：S1.5 给 order 表加了 pay/ship/complete/cancel 时间线列，`orderRow` struct 已加，但 `ListOrders` 的 SELECT 没跟上还在选 13 列。

**修**：复用同文件 `orderTimelineCols` 常量；resp 透传 5 个时间线字段。

### 6. payment-rpc cashier read-after-write（同 cart）

**症状**：下单后立刻进收银台 → "order not found"

**修**：GetCashier 把 SELECT 包到 TransactCtx。

### 7. 收货地址管理页打通

**问题**：`/pages/address/list.vue` 是占位，新用户没默认地址就下不了单。

**实现**：
- 新建 `src/api/address.ts`（list/get/getDefault/add/update/delete/setDefault）
- `pages/address/list.vue` 真接通：列表 + 设默认 + 编辑 + 删除 + 新增
- 新建 `pages/address/edit.vue`：新增/编辑共用，11 位手机号 + 6 字段校验
- `pages.json` 注册 `/pages/address/edit`

### 8. admin-api splitPerms JSON 解析

**问题**：`admin_user.permissions` 列存 JSON 字符串 `'["*"]'`，`splitPerms` 直接 `strings.Split(",")` 当 CSV 切，返回 `permissions:["[\"*\"]"]`。

**修**：先 trim → 检查 `[` 前缀 → `json.Unmarshal`；失败 fallback CSV。

### 9. ⭐ 数据库读写分离架构级修复（方案 A）

**问题深度排查**：cart / order / payment 反复撞 read-after-write 1-2s 延迟，每修一处都要包 TransactCtx，治标不治本。

**根因调查**（详见 `architecture/replication-lag-idle-investigation.md`）：
- 实测 idle lag：slave1 ≈ 273ms / slave2 ≈ 803ms / master2 ≈ 1334ms
- 不是负载，是 `replica-preserve-commit-order=ON` + `sync_binlog=1` + `innodb_flush_log_at_trx_commit=1` + 容器 ZFS fsync 慢的叠加

**生产级方案**（详见 `architecture/read-write-split-option-c-gtid.md`）：
- 方案 A：全 SELECT 走 master（0 成本，MVP 阶段够用）
- 方案 C：GTID 会话一致性（生产正解，7 人天，Sprint 5/6）

**今天的动作**：实施方案 A — `env/mysql/proxysql.cnf` rule_id=2 destination_hostgroup 20→10。

**验证**：5 次连续 e2e（cart add → list / order create → cashier）全部一次通过。
