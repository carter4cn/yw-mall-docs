#### MALL_FIELD_ENCRYPTION_KEY 注入开发态默认

`start.sh` 在 `services_start` 函数前 export 32 字节默认 key（用 `${VAR:-default}` 写法，生产 env 已设则不覆盖）：

```bash
# S4 cryptox: 32 字节 AES key 的开发态默认值。生产部署必须用真实 env 覆盖，
# 否则字段加解密会用这个公开 key — 已存数据将无法迁移。
export MALL_FIELD_ENCRYPTION_KEY="${MALL_FIELD_ENCRYPTION_KEY:-0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef}"
```

**根因**：S4 给 user-rpc 加了 `cryptox.MustInit()` 在 svc 启动时调用，没有 env → 直接 panic。日报 5/15 已经留过这个 TODO。

## ❌ 主动放弃：Redis Sentinel 改造

env baseline 起了 redis-master + 3 个 sentinel（26379-26381），理论上 yw-mall 9 个 RPC 都应该走 sentinel 模式才能利用 failover。但实际检查：

```go
// go-zero v1.10.1 redis/conf.go
RedisConf struct {
    Type  string `json:",default=node,options=node|cluster"`
    ...
}
```

go-zero 的 `RedisConf` 只支持 `node|cluster` 两种类型，**不原生支持 sentinel**。全线改造需要：
1. 加 `mall-common/redisx` 包装 `go-redis` 原生 `NewFailoverClient`
2. 9 个 RPC 的 Redis 调用点全部从 go-zero `redis.Redis` 迁到新包
3. go-zero CachedConn 必须保留 NodeType（CachedConn 跟 sentinel 不兼容），那就是双客户端共存

工作量 3-4 小时 + 增加复杂度，且单机开发场景下 redis-master 单点直连完全够用（sentinel 是 watcher，failover 才推切换）。

**决策**：跳过，留作 Sprint 5/6 专项。已在 commit message 里写明原因。
