# yw-mall MVP 商用 Sprint 计划（6 个月）

**编写日期：** 2026-05-12
**适用阶段：** MVP 试运营（P3 红线 + 部分 P4）
**目标：** 6 个月内接入 50 家店家、日订单 1000，能承接真实交易并满足合规底线
**输入文档：**
- `commercial_readiness_gap.md`（后台差距）
- `platform_completeness_assessment.md`（平台全景差距）

---

## 0. 团队与节奏

### 团队（8 人，按职能分工）

| 角色 | 人数 | 主战场 |
|---|---|---|
| 后端 | 4 | mall-*-rpc / mall-api / mall-admin-api |
| 前端 | 2 | yw-mall-fe (C 端) / yw-mall-admin-fe (admin + merchant) |
| 测试 | 1 | 自动化测试 + 手工回归 |
| DevOps | 1 | CI/CD / 监控 / 部署 / 安全 |

> 若只能 6 人启动：砍掉 1 名前端 + 1 名测试，由后端兼任，整体节奏延后 1–2 个 sprint。

### Sprint 节奏

- **长度**：2 周（10 工作日）
- **总数**：12 个 sprint = 24 周 ≈ 6 个月
- **会议**：周一规划会（2h）、每日 standup（15min）、周五演示 + 复盘（2h）
- **冻结**：每个 sprint 最后 2 天**只修 bug 不接需求**

### 完成定义（DoD，每个 Story 通用）

1. 代码合入 main，PR 至少 1 人 review 通过
2. 单元测试覆盖 ≥ 60%（新代码）
3. 集成测试 / e2e 用例至少 1 条 happy path
4. 部署到 staging 环境跑过
5. Sprint 演示会上演示完整流程
6. 文档（API 注释 / 端点表 / 操作手册）同步更新

## 1. 总体里程碑

```
M1 ──┬── Sprint 1: 支付（微信 JSAPI + Native）
     └── Sprint 2: 支付（支付宝 + 银联 + 收银台 UI）

M2 ──┬── Sprint 3: 退款流程 + 状态机
     └── Sprint 4: 退货/换货 + 财务对账

M3 ──┬── Sprint 5: 安全（MFA + IP 白名单 + 实名）
     └── Sprint 6: CI/CD + 自动化测试基建

M4 ──┬── Sprint 7: 商品 SKU 多规格 + 富文本详情
     └── Sprint 8: 店家 SPA（核心模块 1）

M5 ──┬── Sprint 9: 店家 SPA（核心模块 2）+ 监控告警
     └── Sprint 10: 客服 IM 最小版 + 备份灾备

M6 ──┬── Sprint 11: 库存预警 + 批量工具 + 等保测评准备
     └── Sprint 12: 压测 + 性能优化 + 上线演练 + 切流
```

**累计交付时间表**：

| 里程碑 | 时点 | 关键交付 |
|---|---|---|
| 🟡 **能演示完整购买** | Sprint 2 末（W4） | 用户能下单 → 微信支付 → 看到成功页 |
| 🟡 **能演示完整退款** | Sprint 3 末（W6） | 用户能申请退款 → 店家拒/通过 → 资金到账 |

| 🔴 **正式商用 GA** | Sprint 12 末（W24） | 压测达标 + 等保提交 + 灰度上线 |

## 16. 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| 微信/支付宝资质审批拖延 | 中 | 阻塞 S1-S4 | M0 立即启动；备用方案先用商户号沙箱开发 |

| 等保 2.0 测评不通过 | 中 | 上线 +1 个月 | S5 提前自查；S11 找咨询机构介入 |
| 客服 IM 自研复杂度超估 | 高 | S10 延期 | 优先做"最小可用"版（WebSocket + 单聊 + 历史）；多聊/视频后续做 |
| 关键成员离职 | 低 | S 延期 | knowledge sharing；Pair Programming；文档化 |
| 真实交易后用户投诉爆 | 高 | 名誉风险 | S3-S4 必须做扎实；GA 前先内部用 2 周 |
| 支付通道事故（微信限流） | 中 | 支付成功率下降 | S2 实现支付方式 fallback；监控成功率告警 |
