## 1. 调整说明：为什么把真实支付放到最后？

| 原因 | 影响 |
|---|---|
| **真实通道要花钱**：商户号申请 + 资质审核 + 银行打款验证均有成本 | 不应该让前期工程被资质等待阻塞 |
| **微信/支付宝商户号审批 2-4 周** | 资质拿到前根本无法联调 |
| **风险更可控**：先把退款/退货/对账逻辑跑通再换真实通道 | Mock 阶段可重置数据反复演练 |
| **业务方共识**：先验证全平台其它流程 | 团队对齐 |

### 解决方案：Mock PayChannel

```go
// mall-payment-rpc 抽象不变
type PayChannel interface {
    Pay(ctx context.Context, req *PayReq) (*PayResp, error)
    Refund(ctx context.Context, req *RefundReq) (*RefundResp, error)
    Query(ctx context.Context, orderId string) (*PayStatus, error)
}

// 开发期实现：本地"确认支付/确认退款"按钮立即回写订单
type MockChannel struct { ... }

// S11 替换为 WechatPayChannel / AlipayChannel
```

收银台 UI 在 Mock 模式下展示**显眼"开发模式 - 一键确认支付"按钮**，staging/dev 可用；生产构建强制 `PAYMENT_MOCK_ENABLED=false`（feature flag）。

## 17. 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| 🔴 资质未到 W20 | 中 | S11 阻塞 → GA 推迟 | S0 立即启动；每周同步；微信先开沙箱备用 |
| 🟡 Mock ↔ 真实切换有漏 | 中 | 资金错配 | S11.7 工厂模式 + 大量测试；切换前 staging 全量回归 |
| 🟡 银行对账格式坑 | 高 | S11.8 卡住 | S0 索要账单 sample |
| 🟡 实名 OCR 准确率低 | 中 | 部分认证失败 | S4 加人工兜底；准确率 < 90% 转人工 |
| 🟢 IM 复杂度超估 | 高 | S9 延期 | "最小可用" 限制（单聊 + 文字 + 历史） |
| 🟢 等保整改超期 | 中 | S10 延期 | 提前 S4 自查；S10 找咨询机构 |
| 🟢 关键成员离职 | 低 | 局部 delay | Pair programming + 文档化 |
