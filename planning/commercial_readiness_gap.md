# 距离商用电商后台的差距分析

**编写日期：** 2026-05-12
**版本：** v0.1
**适用范围：** yw-mall 后台（管理员 + 店家）+ 关联 RPC 服务

---

## 0. 摘要

目前 yw-mall 后台已经覆盖了**单店家最小可用闭环**所需的核心模块（身份、商品、订单、评论、投诉、活动、钱包、信用、敏感词等共 47 个 Story 中的约 32 个 P0/P1/P2 Story），但与一个**支撑日订单 ≥ 10 万、店家 ≥ 1000、跨地域运营**的商用电商后台相比，仍有 **80+ 个能力缺口**，可分为：

- **P3 必须补齐才能商用**（运营/财务/合规硬性要求）：约 30 项，预估 4–6 人月
- **P4 商用增强（提效与规模化）**：约 30 项，预估 6–8 人月
- **P5 平台化 / 生态化**：约 25 项，预估 8–12 人月

> ⚠️ 时间估算未包含前端工时、QA、性能/安全压测、文档与培训。商用上线另需 1–2 人月的稳定性建设、监控告警、灰度发布、数据回放演练。

## 5. 当前架构 / 工程债（即时改善项）

这部分**不是新功能**，而是补技术债。

| 类别 | 缺口 | 优先级 |
|---|---|---|
| 部署 | db-init 容器名冲突 workaround，要改 profile | 中 |
| 部署 | 无 K8s manifests；podman-compose 单机部署 | 高（规模化前必做） |
| 部署 | 无 CI/CD；commit 后手工 `podman-compose build && up` | 高 |
| 测试 | 无自动化测试（无 unit、无 integration、无 e2e） | 高 |
| 监控 | 无 Prometheus / 告警 | 高 |
| 文档 | proto 同步 + 客户端封装 流程未脚本化，依赖人工 | 中 |
| 文档 | 没有 API 网关层面的 OpenAPI 规范 | 中 |
| 文档 | 缺少端到端 e2e 测试脚本（C 端 + 后台联调） | 中 |
| 前端 | admin-fe 主 bundle 1.08MB，需 chunk 分包 | 中 |
| 前端 | 无 i18n、无暗黑模式、无 a11y | 低（按市场需求决定） |
| 前端 | merchant SPA 还是占位 | 高（影响店家自助） |
| 安全 | OpLog 只打日志没落库；无 SIEM 链路 | 中 |
| 安全 | JWT 密钥写在 etcd yaml；缺密钥管理（Vault） | 中 |
| 安全 | 后台无 CSRF token / Cookie SameSite 默认 | 低（用 token 不用 cookie 可忽略） |

## 6. 推荐路线图（按业务节奏）

### 阶段 1：**最小商用** —— 6 个月（2 人后端 + 1.5 人前端 + 0.5 测试）
- P3 全部 30 项 ✅
- 工程债"高"项：CI/CD、自动化测试基础设施、监控告警 ✅
- merchant 前端 SPA ✅
- 输出：**可以接 50 家店家试运营**

- P4 营销 + 数据分析 + 物流增强 + 客服工单 ✅
- 引入 K8s + ES 搜索 + CDN ✅
- 输出：**支撑 1000 家店家 / 日单 5 万**

### 阶段 3：**生态化** —— +8 个月（团队扩到 6 后端 + 3 前端 + 2 客户端）
- 多端（小程序 + Native App） ✅
- 开放平台 + 应用市场 ✅
- 数据库分库分表 + 多 AZ ✅
- 输出：**支撑日单 100 万 + ISV 生态**

| 项 | 原因 |

| 区块链溯源 / 跨境数字货币 | 业务方向未确定前不投入 |
| Web3 / NFT 商品 | 同上 |

## 8. 风险预警

| 风险 | 描述 | 应对 |
|---|---|---|
| **资金资损** | 当前没有银行对账、没有提现风控审计、缺少账务台账 → 一旦真有交易很容易丢钱 | P3.4 全部必须做完才能开通真实提现 |
| **数据泄露** | OpLog 已记录，但无人审；JWT 密钥明文配置在 etcd | P3.1 MFA + IP 白名单 + Vault 落地 |
| **法律合规** | 实名认证、个保法用户数据接口、税务报表都缺 | P3.5 全部必做 |
| **性能瓶颈** | 单 MySQL 主、单 Redis、无分库分表，订单表 > 1000 万行 LIKE 会爆 | 阶段 2 引入 ES + 阶段 3 分库分表 |
| **单点故障** | 部署在单 AZ，etcd / MySQL / Redis 都是单实例（虽然有主从但 ProxySQL 自身单点） | 阶段 3 多 AZ 改造 |
| **工程团队效率** | 缺 CI/CD、缺自动化测试，每次发布手工 | 阶段 1 必须解决 |

## 9. 结论

**当前 ≈ 完成度 25–30%**（按商用电商后台总能力的 100% 计）。

按 3 阶段路线图执行，预计：
- **6 个月达到最小商用**（试运营级）
- **10 个月达到规模化运营**（千店级）
- **18 个月达到平台化**（生态级）

完成时间相当激进，假设团队稳定且无重大需求变更。如果需要按"半年最小商用"目标推进，需要立即启动 P3.4（财务）和 P3.5（合规）两条主线 —— 它们工时最长、外部依赖最多（银行通道、第三方风控）。
